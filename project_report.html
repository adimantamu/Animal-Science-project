Daily Master Dataset for Mastitis-Related Analysis

This project consolidates dairy data from multiple farm systems (AFI, sensor logs, and event records) into a single daily per-cow dataset. The entire workflow is implemented in Animal_science_dataset_preparation.ipynb.

What makes this project strong?

The pipeline solves a real-world integration problem: the same biological signal (mastitis risk) is reflected across different systems and sampling rates.

  • Time alignment: everything becomes a daily calendar timeline per cow.
  • Smart patching: missing yield/conductivity are filled using prioritized sources.
  • Behavior + context: rumination/water and management events are attached to each day.

Pipeline (end-to-end execution)

The notebook converts all sources into a consistent daily timeline and merges them using the key (ID, date).

Step 1 — Load AFI base dataset
Read AFI labeled data and keep only the columns needed for joining and labeling: ID, date, DailyYield, Avg.Conductivity, Bacteria, Gram, Type.
Step 2 — Build the daily “skeleton” timeline
For each cow: create a continuous daily range from (min date − 30 days) to (max date + 30 days), then merge the original rows onto this template and fill pathogen metadata.
Step 3 — Patch milk yield & conductivity (priority fill)
Join “before” and “after” milk datasets, convert fields to numeric, and fill missing values using priority: AFI base → before export → after snapshots.
Step 4 — Aggregate rumination and water intake to daily
Concatenate per-cow sensor logs, extract the date from timestamps, then compute mean rumination and total daily water intake per cow/day.
Step 5 — Clean events and attach per-day event lists
Parse the event dates, group by cow/day, strip whitespace, deduplicate and sort event strings, then store them as a list.
Step 6 — Export final dataset
Save the merged master table to Mastitis_Dataset_final.csv.

Yield / conductivity patch logic (as implemented)

The notebook coerces all candidate columns to numeric and then fills missing values in a strict priority order.

final_merge["DailyYield"] = ( final_merge["DailyYield"] .fillna(final_merge["yield_post"]) .fillna(final_merge["yield_after"]) ) final_merge["Avg.Conductivity"] = ( final_merge["Avg.Conductivity"] .fillna(final_merge["cond_post"]) .fillna(final_merge["cond_after"]) )

Daily aggregation logic (rumination + water)

After converting timestamps to dates, the notebook aggregates the sensor stream into a daily table.

rum_water_daily = rum_water_prep.groupby(["ID", "date"]).agg({ "rum_index_seconds": "mean", "water_intake_liter": "sum" }).reset_index()

Final dataset schema & quick facts

This section is meant to be professor-friendly: it explains what each column means and what values to expect.

Column Type Description
ID integer (cow identifier) Unique cow ID used across all joins.
date date Daily calendar date for the observation.
yield numeric Daily milk yield (patched from multiple sources).
conductivity numeric Daily average milk conductivity (patched from multiple sources).
bacteria string (label) Pathogen label associated with the cow cohort (e.g., Staph, SA, MBG, Enterococc).
gram string (label) Gram classification (often P/N depending on label source).
type string (category) Category field from AFI base dataset.
rum_index_seconds numeric Daily rumination index aggregated per cow/day (mean).
water_intake_liter numeric Total daily water intake (liters) aggregated per cow/day (sum).
Event list (string list) Cleaned list of events for that cow/day (deduplicated and sorted).

Notes & best practices (what to mention during evaluation)

These points show you understand real-world data constraints and the trade-offs made in the notebook.

Why the 30-day expansion matters

Building a complete daily timeline per cow makes it possible to compute “days relative to event” features (before/after mastitis, lab results, breeding events) and ensures merges do not silently drop days.

Rumination downsampling trade-off

Filtering timestamps containing '00:00' is a fast way to reduce data size, but it assumes midnight rows are representative or already aggregated. For a strict daily aggregation, you could aggregate all timepoints per day instead.
